# 技术方案对比

## 三套候选方案总览

| 维度 | 方案 A：纯 H5 | 方案 B：安卓原生 | 方案 C：H5 前端 + 后端 |
|---|---|---|---|
| 技术栈 | Vue3 + opencv.js + Tesseract.js + jsPDF | Kotlin + OpenCV SDK + MLKit + iText | Vue3 前端 + Spring Boot + OpenCV(Java) + PaddleOCR |
| 开发周期 | **2~4 周** | 4~8 周 | 3~6 周 |
| 跨平台 | ✅ 所有手机浏览器 | ❌ 仅安卓 | ✅ 前端跨平台 |
| 性能 | 一般（wasm） | **最佳** | 中等（重活在后端）|
| 隐私 | **最佳**（全本地）| **最佳**（全本地）| ⚠️ 需上传图片 |
| 离线可用 | ⚠️ 首次加载后可用 | ✅ | ❌ 需联网 |
| 分发难度 | ✅ 一个链接 | ⚠️ 需装 APK | ✅ 一个链接 |
| 中文 OCR | 较弱（Tesseract）| 中（MLKit 强但要 GMS）| **最强（PaddleOCR）** |
| 开发上手 | **最易** | 中 | 中 |

---

## 方案 A：纯 H5（推荐 MVP 首选）

### 技术栈
```
前端框架：   Vue 3 + Vite + TypeScript
UI：        Vant 4（移动端组件库）
图像处理：   opencv.js（OpenCV 的 wasm 版）
OCR：       Tesseract.js（中文需额外下载语言包）
PDF：       jsPDF
相机：       navigator.mediaDevices.getUserMedia
部署：       静态托管（Vercel / GitHub Pages / 自己的服务器）
```

### 优点
- 一套代码全平台，扫码即用，零安装。
- 全本地处理，证件不上传，隐私最干净。
- 迭代最快，适合个人快速验证。

### 缺点 / 风险
- **opencv.js 体积大（~8MB wasm）**：首屏慢，需异步加载 + CDN。
- **iOS Safari 相机限制多**：必须 HTTPS、`getUserMedia` 兼容性、无法自定义高级拍摄参数。**缓解：先支持「相册选图」，相机作为加分项。**
- **OCR 中文较弱**：Tesseract.js 中文识别率一般，复杂版面易错。

### 适用场景
- 自用、分享给朋友、快速验证产品想法。
- **首版强烈推荐这个**。

## 方案 B：安卓原生 App

### 技术栈
```
语言：       Kotlin
相机：       CameraX
图像处理：   OpenCV Android SDK（native .so）
边缘/透视：  Imgproc.Canny + findContours + getPerspectiveTransform
OCR：       ML Kit Text Recognition（中文）或 Tesseract Android
PDF：       Android PdfDocument 或 iText
架构：       MVVM + Jetpack
```

### 优点
- 性能最好，相机控制精细（对焦、曝光、连拍）。
- 完全离线、无隐私顾虑。
- 可上架应用商店（长期产品化）。

### 缺点 / 风险
- 开发周期最长。
- 只能安卓（iOS 用户用不了）。
- 国内手机无 GMS 时，ML Kit 部分功能受限 → 需退回 Tesseract。

### 适用场景
- 打算长期做产品、只服务安卓用户、追求极致体验。

---

## 方案 C：H5 前端 + 后端处理（OCR 强需求时用）

### 技术栈
```
前端：       同方案 A（去掉前端 OCR）
后端：       Spring Boot（Java）或 FastAPI（Python）
图像处理：   后端 OpenCV（Java: JavaCV / Python: cv2）
OCR：       PaddleOCR（中文识别 SOTA，开源免费）
PDF：       后端 iText / Apache PDFBox
通信：       图片上传 + 返回结构化结果（JSON）+ 处理后图片
部署：       Docker，单台 2C4G 可起步
```

### 优点
- **OCR 中文识别质量最高**（PaddleOCR 是目前开源最强）。
- 前端轻量，老手机也能用。
- 后端可统一升级算法。

### 缺点 / 风险
- **隐私**：证件要上传服务器。缓解：自建 + 加密传输 + 用完即删 + 明确告知。
- 服务器成本（PaddleOCR 推理占内存，建议 ≥2C4G）。
- 必须联网。

### 适用场景
- 对 OCR 准确率要求高、可接受自建服务器的场景。

## 推荐路线（分阶段）

```
阶段 1（MVP，1~2 周）：方案 A 纯 H5
        拍照/选图 → 边缘检测 → 透视校正 → 增强 → 导出 PDF
            │
            ├─ 够用了？→ 停在这里，自用没问题 ✅
            │
阶段 2（+1 周）：加多色彩模式 + 多页 + 证件拼版
            │
阶段 3（按需，+1~2 周）：OCR
        ├─ 中文要求不高 → 前端 Tesseract.js（零成本）
        └─ 中文要求高   → 上方案 C 的 PaddleOCR 后端
            │
阶段 4（可选）：要做正式产品 → 转方案 B 安卓原生重写
```

> 这样可以从「一周内能用」起步，逐步加码，避免一上来就投入过大。
