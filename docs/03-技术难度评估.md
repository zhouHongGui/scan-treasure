# 技术难度评估（逐模块）

> 评级说明：⭐ 简单 / ⭐⭐ 中等 / ⭐⭐⭐ 偏难 / ⭐⭐⭐⭐ 有挑战
> 「能否独立完成」均指：**借助 AI 辅助 + 开源库，一个人可完成**。

---

## 模块 1：相机采集 / 相册选图　⭐~⭐⭐

**技术**
- H5：`navigator.mediaDevices.getUserMedia` 调摄像头；`<input type="file" accept="image/*">` 选相册。
- 安卓：CameraX。

**难点 / 风险**
- iOS Safari 必须在 **HTTPS** 下才给相机权限；`getUserMedia` 在部分内嵌浏览器（微信、QQ）行为不一。
- 拍摄分辨率、对焦需调参，否则糊片。

**缓解**：MVP 先做「相册选图」（100% 兼容），相机作为加分项。

**结论：能独立完成。**

---

## 模块 2：边缘检测（自动找文档四边）　⭐⭐⭐

这是扫描类应用的**第一核心**，决定了「自动」体验。

**技术（OpenCV 经典流程）**
```
灰度化 → 高斯模糊去噪 → Canny 边缘 → findContours 找轮廓
→ 按面积排序 → 取最大四边形 → 得到四个角点
```
- H5：opencv.js 的 `cv.Mat / cv.Canny / cv.findContours`。
- 安卓：OpenCV SDK 的 `Imgproc` 系列。

**难点 / 风险**
- **复杂背景下识别率下降**：桌面纹理、阴影、深色背景上的深色证件。
- **多文档/无明确边缘**：拍书页时书脊会有干扰轮廓。
- **角点顺序**：检测到的 4 点顺序不固定，需排序成 左上→右上→右下→左下，否则透视后上下颠倒。

**缓解（关键设计）**
- 自动检测失败或用户不满意时，**强制提供「手动拖角点」交互**——这是所有扫描 App 的兜底方案，必须有。
- 增加面积/长宽比过滤、凸包处理。

**结论：能独立完成。** 核心算法是 30 年的成熟方案，调参 + 手动兜底即可。

---

## 模块 3：透视校正（摆正）　⭐⭐

**技术**
```
cv2.getPerspectiveTransform(src4pts, dst4pts) → 得到变换矩阵 M
cv2.warpPerspective(img, M, (w,h)) → 输出正视图
```
目标尺寸按 A4 比例（210×297）或证件实际比例。

**难点**
- 角点顺序必须对（依赖模块 2 的排序）。
- 目标宽高的估算：可用四边形边长估算，避免拉伸变形。

**结论：能独立完成。** 纯公式调用。

---

## 模块 4：去阴影 / 图像增强　⭐⭐⭐

拍纸质文档最常见的痛点：**纸张有阴影、光照不均**。

**技术**
1. **去阴影（魔法色彩 / 增亮模式）经典算法**：
   ```
   原图 → 大核高斯模糊得到「背景光照」→ 原图减背景（或除法）→ 归一化
   ```
   即 `enhanced = 255 * (img / blur(img))`，效果惊艳，常被叫做「文档闪光」。
2. **锐化**：Unsharp Masking（高斯模糊差分叠加）。
3. **提亮/对比度**：直方图均衡、CLAHE（局部自适应）。
4. **黑白二值化**：自适应阈值（`ADAPTIVE_THRESH_GAUSSIAN`）或 Otsu，比全局阈值抗光照。

**难点**
- 去阴影的高斯核大小要按图选，太小留指纹级阴影，太大全白。
- 二值化在渐变背景上易出脏块，需局部自适应。

**结论：能独立完成。** 算法成熟，参数微调 1~2 天。

## 模块 5：OCR 文字识别　⭐⭐⭐（中文）/ ⭐⭐（英文）

整个项目里**不确定性最高**的模块。

**技术选型**
| 方案 | 中文效果 | 是否本地 | 成本 |
|---|---|---|---|
| Tesseract.js（前端）| 一般 | ✅ | 免费 |
| Tesseract（后端）| 中等 | ✅ | 免费 |
| **PaddleOCR（后端）** | **最强** | ✅ | 免费 |
| ML Kit（安卓） | 好 | ✅ | 免费（需 GMS）|
| 百度/腾讯 OCR API | 极好 | ❌ | 收费 |

**难点 / 风险**
- **版面分析**：多栏、表格、印章会乱序。
- **手写/低分辨率**：识别率显著下降。
- 前端 Tesseract.js 中文语言包 **~10MB+**，首次加载慢。

**缓解**
- 中文要求不高 → 前端 Tesseract.js，零成本。
- 中文要求高 → PaddleOCR 后端（方案 C），中文准确率业内开源第一梯队。
- 提供 OCR 结果**可编辑**界面，让用户改错字。

**结论：能独立完成，但质量取决于选型。** 建议 MVP 先不做 OCR，或用 Tesseract.js 占位。

## 模块 6：PDF 导出　⭐

**技术**
- H5：`jsPDF`，`addImage(jpeg, 'JPEG', 0, 0, w, h)`。
- 安卓：`PdfDocument` / iText。
- 多页：循环 addPage。

**难点**
- 图片压缩与清晰度平衡（JPEG 质量、DPI）。
- A4 适配：按比例放进 210×297mm 页面。

**结论：能独立完成。** 成熟库直接用。

---

## 模块 7：多页管理 + 证件拼版　⭐⭐

**技术**
- 多页：一个数组维护每页图，缩略图列表，可拖拽排序、删除。
- 证件拼版：Canvas/Bitmap 把身份证正反面按固定坐标贴到 A4 模板上。

**难点**
- 拖拽排序的交互（Vant/Sortable 都有现成组件）。
- 证件尺寸标准：二代身份证 85.6×54mm，需按比例缩放，留白美观。

**结论：能独立完成。**

---

## 模块 8：UI / 交互　⭐⭐

- 移动端组件库（Vant / NutUI）开箱即用。
- 拖角点交互：在 Canvas 上画 4 个可拖拽圆点 + 连线，监听 touch 事件。

**难点**：触摸拖拽要顺滑、Canvas 坐标与图片坐标的换算（DPR 缩放）。

**结论：能独立完成。** 这是体力活，不是技术难点。

# 总体结论：能否独立完成？

## ✅ 能。

| 模块 | 难度 | 自研可行 | 备注 |
|---|---|---|---|
| 相机/选图 | ⭐ | ✅ | iOS 相机是唯一小坑 |
| 边缘检测 | ⭐⭐⭐ | ✅ | 必须配手动调边兜底 |
| 透视校正 | ⭐⭐ | ✅ | 公式调用 |
| 去阴影/增强 | ⭐⭐⭐ | ✅ | 调参 1~2 天 |
| OCR（中文）| ⭐⭐⭐ | ✅ | 选 PaddleOCR 后端最佳 |
| PDF 导出 | ⭐ | ✅ | 成熟库 |
| 多页/拼版 | ⭐⭐ | ✅ | 体力活 |
| UI 交互 | ⭐⭐ | ✅ | 组件库 |

**没有「卡脖子」的模块**——每个难点都有 10 年以上的成熟开源方案。

## 真正的风险（不是技术，是工程）
1. **iOS 浏览器相机适配**：实测机型多，建议先支持相册。
2. **opencv.js 首次加载慢**（~8MB）：需 CDN + 异步加载 + loading 提示。
3. **OCR 中文质量**：开源方案够用但不到商业级，需给用户「可编辑」兜底。
4. **调参耗时**：边缘检测、去阴影在不同场景表现差异大，需要测试集。
5. **精力分配**：一个人做全栈 + UI，建议**严格砍需求**，MVP 先跑通主流程。

## 建议
- **第 1 周就做出能用的 MVP**（拍照→摆正→导出），建立信心。
- OCR、证件拼版、多色彩模式作为后续迭代。
- 不做云同步、协作等复杂功能。
